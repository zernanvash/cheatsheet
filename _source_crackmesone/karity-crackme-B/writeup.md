# karity-crackme-B writeup

Challenge_URL: https://crackmes.one/crackme/6a10641e3fba64e45dceab4f

I finally finished this one. The journey was a bit sweet though, especially that "looks-standard-but-isn't" GF(2^8) table in the middle gave me a real headache. Burned a few hours chasing a wrong assumption. Writing up my notes in case it helps anyone.

## First look

x64 Windows console app, built with MSVC. Nothing weird in the imports, no packer indicators in PEiD/DIE — plain binary. Threw it into IDA and followed `main`.

```c
sub_1400017E0(std::cout, "input: ", envp);
// std::cin >> input
if (v17 == 36) {
    qmemcpy(v15, "*<:", 3);
    while (1) {
        if (*((_BYTE *)v10 + v3 + 7) != (*((_BYTE *)v15 + v3) ^ 0x59))
            break;
        if (++v3 >= 3) {
            v11 = sub_1400011F0(Block);
            ...
        }
    }
}
```

So the input must be exactly 36 bytes. Then it XORs `"*<:"` with `0x59` and compares against bytes 7,8,9 of the input. Quick math:

```
'*' ^ 0x59 = 0x73 = 's'
'<' ^ 0x59 = 0x65 = 'e'
':' ^ 0x59 = 0x63 = 'c'
```

So positions 7..9 must be `"sec"`. Going by the standard CTF flag format `karity{...}`:

```
karity{ sec ?????????????????????????  }
0..6    7..9  10..34                    35
```

Position 35 is almost certainly `}`. 25 unknown bytes in the middle. The real validation happens inside `sub_1400011F0`.

## The cipher (sub_1400011F0)

When it first opened it looked scary, 1500+ lines of pseudo, but the structure cleans up fast once you stare at it. Summary:

```python
state = [0] * 16          # 16-byte internal state, starts zero
key_q1 = 0x026B703D14FC4C9A
key_q2 = 0x5FE21933C7A15581
# little-endian:
# key = [0x9A,0x4C,0xFC,0x14,0x3D,0x70,0x6B,0x02,
#        0x81,0x55,0xA1,0xC7,0x33,0x19,0xE2,0x5F]

for step, ch in enumerate(input):     # 36 iterations
    new_state = [0]*16
    for c in range(16):
        key_byte = key[ key_index(c, step) ]
        new_state[c] = ch ^ gf_mul(state[c], key_byte)
    state = new_state

# After 36 steps, state must equal:
# 0x7CD81D08347C7C08, 0x7C3140977C64EFA8 (little-endian)
```

The multiplication:
```c
v13 = byte_1400033F0[(byte_140005710[a] + byte_140005710[b]) % 0xFF];
```

Looks like a textbook exp/log-table GF(2^8) multiplication. `byte_1400033F0` is the exp table (in `.rdata`), `byte_140005710` is the log table populated at startup by `sub_140001000`:

```c
for (i = 0; i < 255; i++)
    log_table[exp_table[i]] = i;
```

The inner loop's offset arithmetic is written in a slightly obfuscated way. I dug it out of the asm — it writes to 8 different offsets per iteration, each using a different key index. The pattern boils down to:

```python
offsets_order = [0,5,10,15, 4,9,14,3, 8,13,2,7, 12,1,6,11]
o2k = [0]*16
for ki, off in enumerate(offsets_order):
    o2k[off] = ki

def get_key(cell, step):
    return key[ o2k[(cell - 3*step) & 0xF] ]
```

Same 16 key bytes every step, just rotated across the cells. Classic AES-ish permutation.

## First plan: linear system

Multiplication in GF(2^8) distributes over XOR, so the whole cipher above is linear. That means:

- Simulate `karity{sec` forward to get `s9`.
- Walk the target backward through `}` to get `s34`.
- 25 unknown bytes feeding 16 byte equations (128 bits) → linear system.
- Add charset constraints (a-z, 0-9, `_`, `-`) and solve.

I started with Z3 over bitvectors — slow but the logic looked sound. Switched to a GF(2) Gaussian elimination implementation, much faster. Added bit-level constraints since bit7 is always 0 for the charset, bit5 is usually 1 (only `_` has bit5=0), and so on. Reduced the free variables, enumerated, picked a candidate, fed it to the binary:

```
input:
bad!
```

Right. Something is off.

## The trap: the exp table is wrong

The linear math should have worked. My suspicion went to the tables. I generated the "real" GF(2^8)/0x11D exp table from scratch and diffed it against the binary's:

```python
def xtime(a):
    r = a << 1
    if r & 0x100: r ^= 0x11D
    return r & 0xFF

true_exp = []
v = 1
for i in range(255):
    true_exp.append(v)
    v = xtime(v)
```

Result: **195 out of 256 entries differ**. First mismatch at index 60:
- True: `0xB9`
- Binary: `0xB1`

After position 60 the table goes off the rails. There are repeating chunks (e.g. positions 96..108 are a duplicate of 83..95). I tried to back out a single reducing polynomial — at the 59→60 transition it would have to be `0x115`, at 78→79 it would have to be `0x11B` (the AES polynomial). Inconsistent. So this table isn't generated by a clean `xtime` loop at all — it's either hand-edited, intentionally corrupted, or built from a different routine.

Worse: I tested distributivity on the binary's actual `gf_mul`:

```
gf_mul(a^b, c) == gf_mul(a,c) ^ gf_mul(b,c)?
```

Fails on the very first pair I tried. So the operation isn't a field multiplication. **My linear model was wrong from the start.** That explained the bad result.

This is the moment where the challenge really begins. You can't apply Gaussian elimination if XOR doesn't distribute. The 16 byte equations are non-linear functions of the 25 unknowns now.

## Wrong turns

Tried a few things before landing on the working approach:

- **Brute force on the GF(2) residual system** — with the broken table the residual was too large, ~2^54 candidates. No.
- **Pure forward search with backward reachability pruning** — for each cell I precomputed which states at step `k` could still reach `s34[c]` via valid charset inputs. The valid sets saturate to ~256 within 2 backward steps, so pruning only helps for the last 2 steps. Not enough.
- **Meet in the middle** — splitting 25 unknowns evenly gives roughly `38^12 ≈ 10^18` per side. Too big to store, too big to scan.
- **Naive Z3 with the cipher inlined symbolically** — every step does a 256-entry table lookup on a symbolic state. Nest those 25 deep and the expression tree explodes (`O(256^25)`). Z3 chokes immediately.

I almost gave up on Z3 here.

## The encoding that worked

The fix is annoyingly simple in hindsight: **don't let the symbolic expression nest.** Introduce a fresh `BitVec(8)` for the state of each cell at each step, and write the transition as a flat constraint.

```python
# 25 unknown input bytes
inp = [BitVec(f'i{j}', 8) for j in range(25)]
for x in inp:
    sol.add(Or(
        And(x >= 0x61, x <= 0x7a),  # a-z
        And(x >= 0x30, x <= 0x39),  # 0-9
        x == 0x5f, x == 0x2d         # _, -
    ))

# Intermediate state vars: st[k][c] = cell c after processing input[10+k]
st = [[BitVec(f's{k}_{c}', 8) for c in range(16)] for k in range(25)]
```

Each transition is now one level deep. The trick is to build the lookup as a balanced binary tree of `If`s on the input byte — depth 8, ~510 nodes per lookup, no exponential blow-up:

```python
def make_lut(x, table):
    def h(lo, hi):
        if lo == hi: return BitVecVal(table[lo], 8)
        m = (lo + hi) // 2
        return If(ULE(x, m), h(lo, m), h(m+1, hi))
    return h(0, 255)
```

Precompute per-key lookup tables using the binary's actual `gf_mul`:

```python
mul_lut = {}
for k in set(key_bytes):
    mul_lut[k] = [gf_mul(s, k) for s in range(256)]
```

Then the cipher constraints:

```python
# Step 0: from the known s9 state
for c in range(16):
    k = get_key(c, 10)
    sol.add(st[0][c] == inp[0] ^ make_lut(BitVecVal(s9[c], 8), mul_lut[k]))

# Steps 1..24
for k in range(1, 25):
    for c in range(16):
        key = get_key(c, 10+k)
        sol.add(st[k][c] == inp[k] ^ make_lut(st[k-1][c], mul_lut[key]))

# The trailing '}' at step 35:
for c in range(16):
    key = get_key(c, 35)
    sol.add(BitVecVal(target[c], 8) ==
            BitVecVal(0x7D, 8) ^ make_lut(st[24][c], mul_lut[key]))
```

## Takeaways

A few things I want to remember for next time:

The big lesson is to test the algebra you assume. The exp table *looked* standard — first 8 entries are the powers of two you'd expect, the multiplication function has the exact shape of a textbook field op. But assumptions like "this is GF(2^8)/0x11D" or "this distributes over XOR" need a one-line sanity check, otherwise you'll waste an afternoon. A simple `gf_mul(a^b, c) == gf_mul(a,c) ^ gf_mul(b,c)` for random triples would have flagged this in under a second.

For the Z3 encoding, the rule is: keep each constraint shallow, name intermediate values. Substituting a 25-step symbolic state into another 25-step symbolic state is what makes solvers hang. Splitting it into per-step variables turns one impossible problem into 400 trivial ones.
