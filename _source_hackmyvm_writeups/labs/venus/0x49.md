# 0x49

moving from `leona` to `ava`

---

The mission clue pointed toward a specific network service:

```bash
leona@venus:~$ cat mission.txt 
################
# MISSION 0x49 #
################

## EN ##        
User ava plays a lot with the DNS of venus.hmv lately... 
```

I initially tried to query the local DNS server using standard tools like `nslookup` and `dig` against `localhost`. However, the server returned `SERVFAIL`, suggesting it was either misconfigured or restricted.

```bash
leona@venus:~$ nslookup -type=txt venus.hmv localhost
** server can't find venus.hmv: SERVFAIL
```

### The Solution:

Since I couldn't get the information via a network query, I looked for the DNS configuration files directly on the system. On Linux, the **BIND** (Berkeley Internet Name Domain) service stores its zone data in `/etc/bind/`.

By navigating to that directory, I found the zone file for the `venus.hmv` domain.

```bash
leona@venus:~$ cd /etc/bind
leona@venus:/etc/bind$ cat db.venus.hmv
...
ns1     IN      A       127.0.0.1
ava     IN      TXT     [REDACTED]
```

### Explanation:

* **DNS (Domain Name System)**: A system used to translate human-readable domain names into IP addresses. It can also store other types of data, such as text strings.
* **TXT Records**: A type of DNS record that provides text information to sources outside the domain. These are frequently used in CTFs to hide keys or passwords.
* **BIND Configuration**:
* `/etc/bind/named.conf.local`: Usually defines which zones the server is responsible for.
* `db.venus.hmv`: The actual database (zone file) containing the records for that domain.


* **The Finding**: The record indicated that a text query for the subdomain `ava.venus.hmv` would return the string.

Using the string found in the DNS zone file, I successfully logged in as `ava`.

```bash
leona@venus:~$ su - ava
Password: 
ava@venus:~$ id
uid=1050(ava) gid=1050(ava) groups=1050(ava)
```
