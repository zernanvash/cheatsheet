# 0x43

moving from `mercy` to `paula`

---

The mission for this stage gave a subtle hint about user habits:

```bash
mercy@venus:~$ cat mission.txt 
################
# MISSION 0x43 #
################

## EN ##
User mercy is always wrong with the password of paula. 
```

I checked the home directory and noticed that the `.bash_history` file was readable and had a non-zero size. In Linux, this file records the commands previously entered by a user in the terminal.

```bash
mercy@venus:~$ ls -la
total 36
...
-rw-r----- 1 root  mercy  133 Apr  5  2024 .bash_history
-rw-r----- 1 root  mercy   31 Apr  5  2024 flagz.txt   
-rw-r----- 1 root  mercy  190 Apr  5  2024 mission.txt 
```

### The Solution:

When a user accidentally types their password into the terminal instead of a password prompt, it often gets saved into their command history. I inspected the `.bash_history` file to find any leaked credentials.

```bash
mercy@venus:~$ cat .bash_history 
ls -A
ls
rm /
ps
sudo -l
watch tv
vi /etc/logs
su paula
[REDACTED]  <-- Accidentally typed password
history
history -c
logout
...
```

### Explanation:

* **`.bash_history`**: This file stores a list of commands executed by the user. If a user tries to use `su` or `ssh` but fails to wait for the password prompt, they might accidentally paste or type their password directly into the shell, where it is recorded as a command.
* **The Leaked String**: The string appeared immediately after a failed `su paula` attempt, indicating it was likely the intended password.
* **Cleaning Up**: The history also showed an attempt to run `history -c` (clear history), but the password remained in the physical file on the disk.

Using the password found in the history, I successfully logged in as `paula`.

```bash
mercy@venus:~$ su - paula
Password: 
paula@venus:~$ id
uid=1044(paula) gid=1044(paula) groups=1044(paula),1053(hidden)
```
