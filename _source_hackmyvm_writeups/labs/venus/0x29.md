# 0x29

moving from `celeste` to `nina`

---

The mission said:

```bash
celeste@venus:~$ cat mission.txt
################
# MISSION 0x29 #
################

## EN ##
The user celeste has access to mysql but for what?
```

I checked the home directory and, seeing no other leads, I logged into the local MySQL (MariaDB) instance using the credentials provided in the mission hint.

```bash
celeste@venus:~$ mysql -u celeste -p
```

Once inside the database monitor, I explored the available data. I found a database named `venus` containing a table called `people`.

```sql
MariaDB [(none)]> show databases;
MariaDB [(none)]> use venus;
MariaDB [venus]> show tables;
+-----------------+
| Tables_in_venus |
+-----------------+
| people          |
+-----------------+
```

I queried the `people` table to see if it contained any credentials. The table listed several users and their associated "pazz" (passwords).

```sql
MariaDB [venus]> select * from people;
+-----------+---------------+--------------------------------+
| id_people | uzer          | pazz                           |
+-----------+---------------+--------------------------------+
...
|        74 | nina          | [REDACTED]                     |
...
+-----------+---------------+--------------------------------+
```

### Explanation:

* **`show databases;`**: Lists all databases the current user has permission to see.
* **`use venus;`**: Switches the active session to the `venus` database.
* **`select * from people;`**: Retrieves all records from the `people` table, which in this case acted as a credential dump.
* **Identifying the Target**: While many entries contained junk data, row **74** explicitly listed the user `nina` along with her password.

With the password retrieved from the database, I successfully switched to user `nina`.

```bash
celeste@venus:~$ su - nina
Password: 
nina@venus:~$ id
uid=1030(nina) gid=1030(nina) groups=1030(nina)
```
