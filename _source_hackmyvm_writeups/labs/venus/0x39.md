# 0x39

moving from `irene` to `adela`

---

The mission stated:

```bash
irene@venus:~$ cat mission.txt 
################
# MISSION 0x39 #
################

## EN ##        
The user adela has lent her password to irene.
```

In the home directory, I found several files related to encryption and SSH: a private key (`id_rsa.pem`), a public key (`id_rsa.pub`), and an encrypted file named `pass.enc`.

```bash
irene@venus:~$ ls -la
total 44
...
-rw-r----- 1 root  irene 1704 Apr  5  2024 id_rsa.pem 
-rw-r----- 1 root  irene  451 Apr  5  2024 id_rsa.pub 
-rw-r----- 1 root  irene  256 Apr  5  2024 pass.enc   
```

Running `file pass.enc` showed it was raw data, confirming it was likely a binary blob resulting from encryption.

### The Solution:

While I initially tried to use the keys to SSH into `adela@localhost`, the host key verification failed, and the files were clearly intended for a different purpose: **Asymmetric Decryption**.

Since `id_rsa.pem` is a private key and `pass.enc` is an encrypted file, I used `openssl` to decrypt the data. Although the system noted that `rsautl` is deprecated, it still functioned to reveal the cleartext.

```bash
irene@venus:~$ openssl rsautl -decrypt -inkey id_rsa.pem -in pass.enc 
[REDACTED]
```

### Explanation:

* **RSA Encryption**: This is an asymmetric cryptographic algorithm. Data encrypted with a **Public Key** can only be decrypted by the corresponding **Private Key**.
* **`openssl rsautl -decrypt`**:
* **`-inkey id_rsa.pem`**: Specifies the private key used to "unlock" the data.
* **`-in pass.enc`**: The encrypted file containing the secret.


* **The Findings**: The decryption process translated the binary "trash" in `pass.enc` into a human-readable password string.

Using the decrypted password, I successfully logged in as `adela`.

```bash
irene@venus:~$ su - adela
Password: 
adela@venus:~$ id
uid=1040(adela) gid=1040(adela) groups=1040(adela)
```
