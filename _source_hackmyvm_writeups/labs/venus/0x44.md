# 0x44

moving from `paula` to `karla`

---

The mission clue pointed toward a relationship between users and shared permissions:

```bash
paula@venus:~$ cat mission.txt 
################
# MISSION 0x44 #
################

## EN ##
The user karla trusts me, she is part of my group of friends. 
```

I checked my current user identity and discovered that `paula` is a member of a secondary group named `hidden`.

```bash
paula@venus:~$ id
uid=1044(paula) gid=1044(paula) groups=1044(paula),1053(hidden)
```

### The Solution:

Since the mission hinted at a "group of friends," I searched the entire filesystem for files belonging to the `hidden` group that I might have permission to read.

```bash
paula@venus:~$ find / -group hidden -ls 2>/dev/null
   670427      4 -rw-r-----   1 root     hidden         16 Apr  5  2024 /usr/src/.karl-a
```

### Explanation:

* **Group Permissions**: In Linux, files have an owner and an associated group. By being part of the `hidden` group, `paula` has read access to any file where the group permissions are set to `r` (read) and the group owner is `hidden`.
* **`find / -group hidden`**:
* **`/`**: Starts the search from the root directory.
* **`-group hidden`**: Filters for files owned by the `hidden` group.
* **`2>/dev/null`**: Silences "Permission denied" errors for directories `paula` cannot enter, making the output cleaner.


* **The Findings**: The search revealed a hidden file in a system directory: `/usr/src/.karl-a`. The filename itself (`.karl-a`) was a clear pun on the target user, `karla`.

Reading that file revealed the password:

```bash
paula@venus:~$ cat /usr/src/.karl-a
[REDACTED]
```

Using the retrieved password, I successfully logged in as `karla`.

```bash
paula@venus:~$ su - karla
Password: 
karla@venus:~$ id
uid=1045(karla) gid=1045(karla) groups=1045(karla)
```
