# 0x41

moving from `sky` to `sarah`

---

The mission for this stage pointed toward a local web resource:

```bash
sky@venus:~$ cat mission.txt 
################
# MISSION 0x41 #
################

## EN ##        
User sarah uses header in http://localhost/key.php
```

I examined the home directory, but since the hint specifically mentioned a URL and a "header," the objective was clearly to interact with the local web server using `curl`.

```bash
sky@venus:~$ ls -la
total 36
...
-rw-r----- 1 root sky    31 Apr  5  2024 flagz.txt    
-rw-r----- 1 root sky   184 Apr  5  2024 mission.txt 
```

### The Solution:

When accessing the URL normally, the server prompted for a specific condition: `Key header is true?`. This indicated that the PHP script was checking for a custom HTTP header. After several attempts with different header names, I identified the correct key-value pair.

```bash
sky@venus:~$ curl -H "Key: true" http://localhost/key.php
[REDACTED]
```

### Explanation:

* **HTTP Headers**: These are key-value pairs sent along with an HTTP request to provide additional information to the server.
* **Custom Headers**: Developers often use non-standard headers (like `Key`) to gate access to specific data or for internal authentication.
* **`curl -H`**: This flag allows you to manually inject headers into your request.
* **The Logic**: The server was programmed to only release the password if the request included a header named `Key` with the exact value `true`.

Using the retrieved password, I successfully switched to user `sarah`.

```bash
sky@venus:~$ su - sarah
Password: 
sarah@venus:~$ id
uid=1042(sarah) gid=1042(sarah) groups=1042(sarah)
```
