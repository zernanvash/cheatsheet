# 0x34

moving from `noa` to `maia`

---

The mission said:

```bash
noa@venus:~$ cat mission.txt
################
# MISSION 0x34 #
################

## EN ##
The password of maia is surrounded by trash
```

I checked the home directory and found a file aptly named `trash`. This file is significantly larger than the others (3818 bytes) and contains binary data.

```bash
noa@venus:~$ ls -la
total 36
...
-rw-r----- 1 root noa   31 Apr  5  2024 flagz.txt
-rw-r----- 1 root noa  159 Apr  5  2024 mission.txt
-rw-r----- 1 root noa 3818 Apr  5  2024 trash
```

Running `file trash` confirmed it is identified as "data," meaning it's not a standard text file and contains many non-printable characters.

```bash
noa@venus:~$ file trash
trash: data
```

### The Solution:

To find the password "surrounded by trash," I used the `strings` command. This tool ignores the binary "junk" and extracts sequences of printable characters.

```bash
noa@venus:~$ strings trash
...
hl(p
fEr:
OdBb
[REDACTED]  <-- Potential Password
...
```

### Explanation:

* **`strings`**: This is the go-to tool when a password or flag is hidden inside a binary, executable, or corrupted file. It looks for sequences of 4 or more printable characters.
* **The Findings**: Amidst the random snippets of text, one specific alphanumeric string stood out as a likely candidate for a password. In this case, the string was `[REDACTED]`.

Using the extracted string from the `trash` file, I successfully switched to user `maia`.

```bash
noa@venus:~$ su - maia
Password: 
maia@venus:~$ id
uid=1035(maia) gid=1035(maia) groups=1035(maia)
```
