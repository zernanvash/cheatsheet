The mission clue for this level was found within the files retrieved from the FTP server:

```bash
asteria@hades:/tmp/ftpp$ cat mission.txt  
################
# MISSION 0x13 #
################

## EN ##
The user atalanta has done something with our account. 
```

### The Solution: FTP Pivot & Local Write Permissions

After discovering the password for `astraea` in the previous challenge, I encountered a block when trying to log in via SSH. The system allowed the password but immediately terminated the connection.

#### Step 1: SSH Attempt (Connection Closed)

Standard shell access was restricted for this user, likely through a configuration like `/usr/sbin/nologin`.

```bash
asteria@hades:~$ ssh astraea@localhost
astraea@localhost's password: 
Connection to localhost closed.
```

#### Step 2: Preparing a Writable Workspace

When using FTP to download files with `mget`, the FTP client tries to save the files into your **current local directory**. Because the current directory (`/pwned/asteria`) was not writable, I had to create a temporary workspace in `/tmp`.

```bash
asteria@hades:~$ mkdir /tmp/ftpp 
asteria@hades:~$ cd /tmp/ftpp
```

#### Step 3: FTP Login and File Retrieval

From the new writable directory, I connected to the local FTP server.

```bash
asteria@hades:/tmp/ftpp$ ftp astraea@localhost
Connected to localhost.
220 (vsFTPd 3.0.3)
Name (localhost:asteria): astraea
331 Please specify the password.
Password: 
230 Login successful.
```

I used `mget *` to download the target user's password file, the mission details, and the flag.

```bash
ftp> ls
-rw-r-----    1 0        2004           21 Jan 17 11:00 atalanta.txt
-rw-r-----    1 0        2004           22 Apr 05  2024 flagz.txt
-rw-r-----    1 0        2004          181 Apr 05  2024 mission.txt

ftp> mget *
```

### Explanation

* **Restricted Shells**: Users can be barred from SSH but still granted access to other services like FTP for file management.
* **FTP Local Permissions**: A common mistake when using the `ftp` command is forgetting that you must have **write permissions** in the directory where you launched the FTP client. Without a writable local path, `get` or `mget` commands will fail.
* **Credential Harvesting**: The password for the next user was stored in `atalanta.txt`.

### Execution

Once the transfer was complete, I exited FTP and read the password for the next level.

```bash
asteria@hades:/tmp/ftpp$ cat atalanta.txt 
[REDACTED]
```

Using this credential, I successfully established a full SSH session as **atalanta**.

```bash
asteria@hades:/tmp/ftpp$ ssh atalanta@localhost
...
atalanta@hades:~$ id
uid=2005(atalanta) gid=2005(atalanta) groups=2005(atalanta)
```
