# 0x17

moving from `aegle` to `calliope`

---

The mission for this stage suggests that user **calliope** has information or files that need to be "looked at":

```bash
aegle@hades:~$ cat mission.txt 
################
# MISSION 0x17 #
################

## EN ##
User calliope likes to have her things looked at.
...
```

In the current directory, a file named `calliope_pass.txt` appeared to be the target, but its permissions were restricted to `root` and the group `calliope`.

```bash
-rw-r----- 1 root  calliope   21 Apr  5  2024 calliope_pass.txt
```

### 1. The Obstacle: Directory Permissions

Checking `sudo -l` revealed that user `aegle` could run the `cat` command as user `calliope` without a password:

```bash
aegle@hades:~$ sudo -l
...
User aegle may run the following commands on hades:
    (calliope) NOPASSWD: /bin/cat
```

However, attempting to read the local password file failed even when acting as Calliope:

```bash
aegle@hades:~$ sudo -u calliope /bin/cat ./calliope_pass.txt 
/bin/cat: ./calliope_pass.txt: Permission denied
```

**Why it failed:** As you noted, this was caused by the **parent directory permissions** of Aegle's home.

* The directory `/pwned/aegle` has permissions `drwxr-x---`.
* User `calliope` is not the owner and not in the `aegle` group, so she falls into the "others" category (the final `---`).
* In Linux, to access *any* file inside a directory, a user must have **Execute (x)** permissions on the parent directory to "traverse" or enter it. Because `calliope` cannot traverse `/pwned/aegle`, the `cat` command (running as her) cannot reach the file.

### 2. The Solution: Retrieving the Flag for the Website

Since I could run `cat` as `calliope`, I needed to find a location where she actually had the permissions to read her own files. I located her home directory.

```bash
aegle@hades:~$ find / -name "*calliope*" 2>/dev/null
/pwned/calliope
...
```

By using the privileged `cat` command, I was able to read the flag from Calliope's own home directory and redirect the output to the terminal.

```bash
aegle@hades:~$ sudo -u calliope /bin/cat /pwned/calliope/flagz.txt > /dev/stdout
[REDACTED]
```

### 3. Explanation

* **Directory Traversal**: This challenge highlights that file-level permissions are irrelevant if the parent directory blocks access (missing `x` bit).
* **External Integration**: This level uniquely integrates the laboratory with the HackMyVM website. The local flag retrieved from the system is not the password itself, but a token.
* **The Finding**: By submitting the retrieved flag to the `/hades/flagz.php` page on the website, the platform provided the actual login password for user **calliope**.

### 4. Execution

After retrieving the password from the website, I successfully established a full SSH session.

```bash
aegle@hades:~$ ssh calliope@localhost
...
calliope@hades:~$ id
uid=2009(calliope) gid=2009(calliope) groups=2009(calliope)
```
