# 0x02

moving from `acantha` to `alala`

---

The mission for this stage involved interacting with a custom program to retrieve a password:

```bash
acantha@hades:~$ cat mission.txt 
################
# MISSION 0x02 #
################

## EN ##        
The user alala has left us a program, if we insert the 6 correct numbers, she gives us her password!
```

In the home directory, I found an executable binary named `guess`.

```bash
acantha@hades:~$ ls -la
total 48
...
-rw-r-x--- 1 root    acantha 16064 Apr  5  2024 guess
-rw-r----- 1 root    acantha   275 Apr  5  2024 mission.txt
```

### The Solution:

The program requires a 6-digit PIN code to reveal the password. Instead of attempting to brute-force the numbers, I analyzed the binary's content using the `strings` command to see if any sensitive information was embedded in the compiled code.

```bash
acantha@hades:~$ strings ./guess
...
Enter PIN code:
[REDACTED]
NO :_(
...
```

### Explanation:

* **Static Analysis**: Using `strings` is a form of basic static analysis. It extracts sequences of printable characters from binary files. In many beginner CTF challenges, "secrets" or "passwords" that the program is supposed to print are stored as plain text within the binary's `.rodata` (read-only data) section.
* **Hardcoded Credentials**: The string stood out as it was positioned right between the "Enter PIN code" prompt and the failure message. This is a clear indicator of a hardcoded string that the program likely prints upon a successful "guess."
* **The Findings**: The string identified in the binary was the actual password for the user **alala**.

### Execution:

Using the identified password, I successfully logged in via SSH to establish a session as `alala`.

```bash
acantha@hades:~$ ssh alala@localhost
...
alala@hades:~$ id
uid=2044(alala) gid=2044(alala) groups=2044(alala)
```
